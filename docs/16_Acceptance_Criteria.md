# 16. Acceptance Criteria

## Мета документа

Цей документ визначає критерії прийняття для функціональностей ERP-системи, які повинні бути виконані для підтвердження того, що рішення відповідає бізнес-потребам користувачів.

Acceptance Criteria сформовані на основі User Stories та Use Cases.

---

# AC-01. Відображення доступності продукції

## Пов'язано з:

* User Story 1 — Прогноз доступності продукції для продажу
* UC-01 — Перевірка доступності продукції

## Критерії прийняття:

* Користувач може переглянути поточну кількість готової продукції на складі.
* Користувач бачить продукцію, яка знаходиться у виробничому процесі.
* Система відображає прогнозований строк готовності наступної партії.
* Дані оновлюються відповідно до фактичного стану виробництва.
* Інформація доступна без використання зовнішніх таблиць або ручного збору даних.

---

# AC-02. Контроль критичного рівня складу

## Пов'язано з:

* User Story 2 — Контроль наповнення складу готової продукції
* UC-02 — Контроль заповнення складу пелет

## Критерії прийняття:

* У системі можна встановити мінімальний та критичний рівень запасу.
* Система відображає поточний рівень заповнення складу.
* При досягненні критичного рівня відповідальний користувач отримує повідомлення або візуальний сигнал.
* Користувач може визначити необхідність початку продажу продукції.

---

# AC-03. Формування пропозиції клієнту

## Пов'язано з:

* User Story 3 — Планування машинного відвантаження
* UC-03 — Формування пропозиції клієнту

## Критерії прийняття:

* Менеджер з продажу бачить доступний для продажу обсяг продукції.
* Система дозволяє врахувати мінімальний економічно доцільний обсяг відвантаження.
* Менеджер може сформувати пропозицію клієнту на основі актуальних даних.
* При недостатній кількості продукції система дозволяє визначити очікуваний строк готовності.

---

# AC-04. Планування транспортного відвантаження

## Пов'язано з:

* User Story 6 — Планування відвантаження клієнту
* UC-04 — Планування транспортного відвантаження

## Критерії прийняття:

* Система дозволяє передати інформацію про замовлення логісту.
* Враховується тип транспорту та його обмеження.
* Визначається можливий обсяг завантаження автомобіля.
* Інформація про транспортні витрати може бути додана до замовлення.
* Статус доставки доступний відповідальним користувачам.

---

# AC-05. Контроль забезпечення пакуванням

## Пов'язано з:

* User Story 5 — Контроль забезпечення виробництва пакувальними матеріалами
* UC-05 — Контроль забезпечення виробництва пакуванням

## Критерії прийняття:

* Система відображає залишки мішків та біг-бегів.
* Визначається мінімальний запас пакувальних матеріалів.
* При досягненні мінімального рівня формується потреба у закупівлі.
* Менеджер закупівель отримує інформацію для планування закупівлі.

---

# AC-06. Прогнозування виробництва пелет

## Пов'язано з:

* User Story 4 — Врахування виробничого прогнозу
* UC-06 — Аналіз виробничого прогнозу

## Критерії прийняття:

* Система дозволяє пов'язати основне виробництво з утворенням відходів.
* Враховується тип продукції основного виробництва.
* Можна оцінити потенційний обсяг сировини для виробництва пелет.
* Дані можуть використовуватися для планування майбутніх продажів.

---

# AC-07. Контроль статусу замовлення

## Пов'язано з:

* User Story 6 — Планування відвантаження клієнту
* UC-07 — Відстеження статусу замовлення

## Критерії прийняття:

* Кожне замовлення має поточний статус.
* Статуси відображають основні етапи процесу:

  * створено;
  * підтверджено;
  * продукція готується;
  * готово до відвантаження;
  * відвантажено.
* Користувачі мають доступ до актуальної інформації відповідно до їх ролей.

---

# AC-08. Достовірність управлінської інформації

## Пов'язано з:

* User Story 7 — Аналіз ефективності процесу

## Критерії прийняття:

* Дані у системі відповідають фактичному стану процесів.
* Керівники можуть отримувати інформацію для прийняття управлінських рішень.
* Дані можуть використовуватися для формування KPI.
* Всі ключові показники мають визначене джерело даних.

---

## Definition of Done

Функціональність вважається прийнятою, якщо:

* усі визначені Acceptance Criteria виконані;
* користувачі можуть виконувати необхідні операції без ручного дублювання інформації;
* дані відповідають бізнес-процесу;
* результат пройшов перевірку ключових стейкхолдерів.
